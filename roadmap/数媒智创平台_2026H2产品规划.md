## 一、现状盘点

### 1.1 已建成能力

基于 PRD V1.0，平台已建成"主站 + 三个子系统"的基础架构：

| 系统 | 地址 | 核心能力 |
|------|------|---------|
| 主站 | x.sztv.com.cn | AI对话工作台、热点榜单、历史对话、个人中心 |
| 创作子站 | geekai.sztv.com.cn | 图片生成、视频创作、智能配音、数字主播、实用工具 |
| 知识库子站 | weknora.scms.sztv.com.cn | 企业级RAG知识库管理（WeKnora） |
| 技能市场 | SkillHub | AI技能发布、搜索、订阅 |

支持7个大模型切换，预置多个媒体场景智能体，集成8个第三方工具入口，已有热点助手原型（COS托管）。

### 1.2 核心痛点

| # | 痛点 | 影响 |
|---|------|------|
| 1 | **模块各自为阵** | 主站、geekai、WeKnora、SkillHub 四子系统割裂，用户需手动跳转，体验断裂 |
| 2 | **有人用但留不住** | 用户做完一次任务就走，缺乏持续使用动力和个人沉淀 |
| 3 | **热点无法贯通生产** | 热点榜单只能"看"，无法直接进入创作流，热点助手引进后形成孤岛 |
| 4 | **无法量化平台价值** | 缺乏数据看板，领导看不到使用数据，无法证明平台价值，影响资源争取 |

**痛点关联：** 模块割裂(1) → 热点无法贯通(3) → 体验断裂 → 留不住(2)；而没有数据(4) → 无法证明价值 → 无法争取资源 → 上述问题更难解决。**#4 是破局点，#1+#3 是产品核心抓手。**

### 1.3 已知技术风险

| 风险 | 现状 | 必须解决时间 |
|------|------|------------|
| WeKnora SSO 登录失败 | need-login 错误，跨域Cookie问题 | Q3（知识库能力化前置） |
| 子站登录态丢失 | 切换子站后主站被登出 | Q3（统一用户系统） |
| 智能体接口不稳定 | 热搜等任务执行失败 | Q3（重试+降级+缓存） |
| AIGC内容合规 | 无审核机制 | Q4（融入全链路审核） |

## 二、规划目标

### 2.1 业务目标

1. **证明平台价值**：建成数据看板，用数据说话，让领导看到平台的使用规模与效率提升
2. **打通全链路**：以热点为入口，贯通"发现→分析→创作→发布→审核"的内容生产流水线
3. **沉淀平台能力**：将知识库、AIGC图片生成从子系统升级为可复用的平台能力（API化）
4. **提升用户体验**：界面升级 + 流程优化，提高用户留存

### 2.2 量化指标（建议）

| 维度 | 指标 | 目标 |
|------|------|------|
| 活跃 | 月活用户数（MAU） | 看板上线后持续观测，H2环比增长30% |
| 使用 | 日均生成次数（图文+视频+配音） | 看板上线后持续观测 |
| 链路 | 热点→创作转化率 | Q4全链路上线后，热点详情页"生成"按钮点击率≥15% |
| 留存 | 周留存率 | H2末达到40%以上 |
| 效率 | 单篇图文平均生产耗时 | 较无平台时缩短50%以上 |

> 注：以上指标需在数据看板上线后建立基线，再逐步校准目标值。

## 三、规划方向

### 3.1 方向总览

基于团队资源（3人）和预算约束（无预算，依靠自研+开源），H2 聚焦以下六大方向：

| # | 方向 | 优先级 | 定位 | H2目标 |
|---|------|--------|------|--------|
| 1 | 热点全链路 | **P0** | 业务核心 | 热点→AIGC→发布 全链路打通 |
| 2 | 知识库能力化 | P1 | 能力层 | 从子系统变为API能力，试点对接 |
| 3 | AIGC图片生成优化+能力化 | P1 | 能力层 | 优化生成体验，API化 |
| 4 | 统一用户系统+数据看板 | **P0** | 技术底座 | SSO修复+埋点+看板上线 |
| 5 | 内容安全审核 | P1 | 合规保障 | 调研+基础审核流程 |
| 6 | 移动端单能力H5 | P2 | 场景延伸 | 调研选型，H2不出活 |

**明确排除（H2不考虑）：** 会员权益体系、企业组织权限、技能市场激励。

### 3.2 分层架构

```
┌───────────────────────────────────────────────────┐
│              业务层                                │
│   热点全链路：发现 → 分析 → 创作 → 发布 → 审核    │
├────────────────┬──────────────────────────────────┤
│   能力层        │                                  │
│  知识库能力     │  AIGC图片生成能力                 │
│  (API/SDK)     │  (API/SDK)                       │
├────────────────┴──────────────────────────────────┤
│              底座层                                │
│   统一用户系统 + 行为埋点 + 数据看板               │
└───────────────────────────────────────────────────┘
```

**执行原则：** 底座先行（方向4）→ 能力并行（方向2+3）→ 业务串联收口（方向1）

## 四、方向详细说明

### 4.1 方向1：热点全链路（P0）

**定位：** 以热点为入口的内容生产流水线，这是平台区别于"工具合集"的核心。

**已有基础：**
- 热点系统原型已开发（COS托管），含列表页和详情页
- 详情页具备：热度评级(S+/S/A/B)、关键时间线、子事件、24h热度趋势图、相关文章聚合、AI事件摘要
- 详情页已有"仿写/生成/快报"三个AI入口（当前未打通AIGC生产）

**Q3 目标：热点系统正式化 + AIGC对接**

| 任务 | 说明 | 产出 |
|------|------|------|
| 原型正式部署 | 从COS迁移至平台内部署，接入统一用户系统 | 热点系统可在平台内访问 |
| 生产需求优化 | 根据编辑/记者实际使用反馈优化交互和功能 | 优化版热点列表+详情页 |
| AIGC对接-图文 | 详情页"生成"按钮 → 打通图文创作流程 | 热点→AI写稿→图片生成→图文成品 |
| 智能体实现路径 | 确定智能体串联方案（即使不能立即实现） | 技术方案文档 |

**全链路定义（Q3 MVP）：**
```
热点发现 → 事件详情 → AI事件摘要(已有) → 点击"生成" 
→ AI写稿(智能体/对话) → 图片生成(geekai) → 图文成品
```

**Q4 目标：全链路闭环到发布**

| 任务 | 说明 | 产出 |
|------|------|------|
| 对接发布平台 | 打通IMP/XGC等CMS系统，支持内容导入 | AIGC成品可一键导入发布平台 |
| 审核环节融入 | 发布前内容审核（敏感词+人工复核） | 审核流程嵌入发布链路 |
| 全链路串联 | 热点→创作→导入→审核→发布 完整流程 | 端到端可用流程 |

**全链路定义（Q4 闭环）：**
```
热点发现 → 事件详情 → AI分析 → AIGC图文创作 
→ 成品产出 → 导入发布平台(IMP/XGC) → 内容审核 → 发布
```

**技术方案路径（智能体串联）：**

短期（Q3）：按钮跳转 + 上下文传递
- 详情页"生成"按钮携带事件摘要作为提示词，跳转到AI对话或图片生成页面
- 用户手动确认和调整后生成

中期（Q4）：智能体编排
- 通过SkillHub技能编排，实现"一键生成图文"自动化流程
- 智能体自动调用：事件摘要 → 写稿 → 配图 → 组装图文

长期（2027H1）：全自动热点跟写
- 监控热点异动 → 自动触发内容生产 → 人工审核后发布

### 4.2 方向2：知识库能力化（P1）

**定位：** 从"WeKnora iframe嵌入主站"升级为"可被任意系统调用的知识库API能力"。

**现状：**
- WeKnora作为独立子系统，通过iframe嵌入主站
- SSO登录存在异常（need-login错误）
- 知识库能力封闭在子站内，无法被其他系统使用

**Q3 目标：启动能力化 + 试点对接**

| 任务 | 说明 | 产出 |
|------|------|------|
| SSO修复 | 排查跨域Cookie与SSO接口，修复登录链路 | 主站可正常单点进入知识库 |
| 能力梳理 | 梳理知识库核心能力：文档解析、RAG检索、智能问答 | 能力清单+API设计文档 |
| API封装（基础） | 将核心能力封装为REST API | 知识检索API、知识问答API |
| 试点对接 | 与OA系统或其他生产系统对接验证 | 1个场景落地验证 |

**Q4 目标：持续优化 + 扩展对接**
- 根据试点反馈优化API
- 扩展对接更多系统
- 支持更丰富的文档类型和检索策略

**技术路线（基于开源，无预算）：**
- 基于现有 WeKnora（开源）进行改造，不购买商业RAG服务
- 封装 RESTful API，提供知识检索、知识问答、文档上传等接口
- 通过 API Key + 统一用户系统进行鉴权

### 4.3 方向3：AIGC图片生成优化 + 能力化（P1）

**定位：** 优化图片生成体验，并抽取为API能力供其他系统调用。视频能力当前较弱，H2优先聚焦图片。

**现状：**
- 图片生成模型：Nano-Banana
- 参数较基础：质量标准、尺寸1024x1024、样式生动
- 最多上传5张参考图，单张≤5MB
- 功能封闭在geekai子站内

**Q4 目标（Q3不投入主力，Q4视进度推进）：**

| 任务 | 说明 | 产出 |
|------|------|------|
| 生成体验优化 | 更多模型选项、更多尺寸/风格、批量生成 | 优化版图片生成 |
| API封装 | 图片生成能力封装为API | 图片生成API |
| 热点链路集成 | 热点详情页可直接调用图片生成能力 | 热点→配图一键生成 |

**技术路线：**
- 图片生成API与知识库API采用统一的API网关和鉴权机制
- 支持异步生成 + 回调通知
- 生成结果统一存储，支持历史查看和下载

### 4.4 方向4：统一用户系统 + 数据看板（P0）

**定位：** 技术底座，先行建设，支撑上层所有业务方向。

#### 4.4.1 统一用户系统

**要解决的问题：**
- WeKnora SSO 登录失败
- 切换子站后主站登录态丢失
- 各子系统独立账号，无法统一管理

**Q3 目标：**

| 任务 | 说明 | 产出 |
|------|------|------|
| 统一认证中心 | 建立统一SSO认证服务 | 主站+各子站共享登录态 |
| 子站接入 | geekai、WeKnora、SkillHub 接入统一认证 | 切换子站无需重新登录 |
| 用户信息统一 | 用户基本信息、权限标识统一管理 | 统一用户数据结构 |

**技术路线（自建，无预算）：**
- 基于 OAuth 2.0 / CAS 协议自建轻量SSO服务
- 各子系统通过 Token 共享登录态
- 不购买商业身份认证服务

#### 4.4.2 数据看板

**要解决的问题：** 领导要数据，平台无埋点无看板。

**Q3 目标：第一层看板上线**

| 任务 | 说明 | 产出 |
|------|------|------|
| 埋点方案设计 | 设计用户行为埋点规范 | 埋点文档 |
| 主站埋点实施 | 在主站关键路径加入埋点 | 主站行为可追踪 |
| 看板开发 | 开发数据统计看板 | 第一版看板上线 |

**第一层看板指标：**

| 分类 | 指标 |
|------|------|
| 用户规模 | 注册用户数、DAU、MAU、新增用户 |
| 功能使用 | 各功能模块使用次数、使用人数、使用排行 |
| 用户路径 | 日均会话数、平均会话深度、核心路径转化漏斗 |
| 内容生产 | 日均生成次数（图文/视频/配音/数字主播）、生成成功率 |
| 系统运行 | 接口响应时长、错误率、模型调用量 |

**技术路线（自建，无预算）：**
- 埋点：前端轻量埋点SDK（自研）+ 后端日志聚合
- 看板：前端图表组件（ECharts/Chart.js）+ 后端数据聚合API
- 不购买BI工具

### 4.5 方向5：内容安全审核（P1）

**定位：** 合规保障，融入全链路发布环节，不独立做系统。

**现状：** 当前无任何内容审核机制，AIGC产出存在版权和合规风险。

**Q4 目标：基础审核流程上线**

**三层审核策略：**

| 层级 | 时机 | 方案 | 成本 |
|------|------|------|------|
| 事前 | 用户输入侧 | 敏感词过滤（开源词库） | 低，自建 |
| 事后 | 成品产出侧 | 内容审核 + 人工复核队列 | 中，自建流程 |
| 溯源 | 发布侧 | 内容水印 + 审核记录留档 | 低，自建 |

**技术路线（开源+自建，无预算）：**
- 敏感词检测：使用开源敏感词库（如 mint-filter / sensitive-word）
- 不购买天御/易盾等云审核服务
- 人工复核：在看板中增加审核队列，编辑人员人工确认
- 水印：图片/视频自动添加平台水印

### 4.6 方向6：移动端单能力H5（P2）

**定位：** 场景延伸，非核心链路，H2不出活，2027 H1推进。

**思路：** 不做全站适配，做"单能力H5页面"，供编辑/记者外出使用。

**候选H5能力：**

| H5页面 | 核心功能 | 使用场景 |
|--------|---------|---------|
| 热点速览 | 查看实时热点、收藏 | 外出时跟踪热点 |
| AI写稿 | 输入主题→AI生成稿件 | 突发新闻现场快速出稿 |
| 智能配音 | 文本→语音生成+下载 | 外出剪辑配音 |
| 创作历史 | 查看+下载已生成内容 | 随时获取作品 |

**部署方式：** 独立H5页面，可通过企业微信/飞书工作台入口访问。

| Q3三线并行，人手不足 | 交付延期 | 高 | 知识库Q3只做轻量试点，不投入重兵 |
| IMP/XGC接口不开放 | 全链路终点断裂 | 中 | Q3第1周摸底，若不开放则改用人工导入 |
| 热点数据源不稳定 | 热点系统无数据 | 中 | 确认数据源方案，准备备用数据源 |
| 领导临时插入需求 | 打乱排期 | 中 | 规划保留弹性，每月预留1周缓冲 |
| WeKnora SSO修复困难 | 知识库能力化阻塞 | 中 | 若2周内无法修复，提供独立登录入口作为过渡 |
| 无预算限制审核能力 | 内容安全方案受限 | 高 | 使用开源方案，不依赖商业服务 |

### 6.3 弹性预留

- 每月预留约1周时间应对领导临时需求和紧急bug修复
- 方向6（移动端H5）作为可随时启动的备选方向，视Q3进度决定是否Q4提前启动
- 如Q3进度超前，可在Q4增加图片生成优化（方向3）的投入
